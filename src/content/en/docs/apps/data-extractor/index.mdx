# CD2030 DHIS2 Data Extractor

The **CD2030 DHIS2 Data Extractor** helps teams connect to DHIS2, prepare metadata, map indicators, and download routine data in a way that is more stable and easier to reproduce than manual extraction.

It is designed to support large, multi-period downloads while reducing server strain, avoiding unnecessary repeat requests, and preserving progress through local caching.

<Callout type="info">
   **Access disclaimer:** The CD2030 DHIS2 Data Extractor does not allow unauthorized access to DHIS2 data.
</Callout>

## What the extractor does

The extractor supports a practical end-to-end workflow:

1. Sign in to a DHIS2 instance with valid user credentials.
2. Read and cache metadata needed for extraction.
3. Map source metadata to the required analytical structure.
4. Download routine data in manageable requests.
5. Reuse cached metadata and completed download chunks to minimize repeated work.

This process helps analysts produce a cleaner and more reproducible starting point for downstream analysis in DataSuite.

## Step 1: Log in to DHIS2

The first step is to connect the application to the relevant DHIS2 server. Users provide the DHIS2 instance URL together with their username and password. Once authenticated, the extractor can read only the metadata and data that the logged-in user is already authorized to access in that DHIS2 environment.

![DHIS2 login screen](/images/extractor_login.jpg)

![DHIS2 server URL login screen](/images/extractor_login_url.jpg)

<Callout type="info">
  **Important:** Access rights are controlled by DHIS2, not by the extractor. If a user cannot see a dataset, organisation unit, or indicator in DHIS2, the extractor will not bypass those permissions.
</Callout>

Why this matters:

- It confirms the extractor is working against the correct DHIS2 instance.
- It ensures all subsequent mapping and download steps use the user's approved permissions.
- It avoids offline template matching against the wrong server configuration.

<Callout type="info">
  **Tip:** Start by confirming the DHIS2 base URL before entering credentials. A correct URL prevents login failures caused by pointing to the wrong server environment.
</Callout>

<a id="step-2-read-and-cache-metadata"></a>

## Step 2: Read and cache metadata

After login, the extractor retrieves the metadata needed to understand the structure of the DHIS2 instance. This can include items such as organisation units, data elements, indicators, category option combinations, datasets, and other configuration objects required for mapping and extraction.

Rather than downloading the same metadata repeatedly for every step, the extractor caches it locally for reuse during the session and in later work where appropriate.

<Callout type="info">
  **Why caching helps:** Metadata caching reduces repeated calls to DHIS2, speeds up setup for later runs, and creates a smaller download footprint because the extractor only needs to refresh what is necessary.
</Callout>

<Callout type="warning">
  **Check before proceeding:** If the metadata in DHIS2 has changed significantly since the last session, users should refresh the metadata cache before final extraction so mapping decisions reflect the latest configuration.
</Callout>

<a id="step-3-map-metadata"></a>

## Step 3: Map metadata

Mapping is the stage where DHIS2 source metadata is linked to the structure needed for downstream analysis. The extractor supports two mapping approaches depending on the use case.

### Countdown option

The **Countdown** option maps indicators and related metadata to the **Countdown harmonized MNCH data template**. This is the preferred approach when teams are preparing data for Countdown to 2030 workflows and want a standardized structure that aligns with the embedded resource template used by the application.

This option is useful when:

- the analysis is focused on harmonized MNCH indicators,
- teams want consistency across countries or time periods,
- outputs are intended for Countdown-aligned analytical workflows.

<Callout type="info">
  **Recommended for standardization:** Use the Countdown option when your goal is to produce data aligned with the harmonized MNCH template embedded in the extractor resources.
</Callout>

### Custom option

The **Custom** option allows users to create mappings for other programs or use cases that do not fit the Countdown harmonized template. This gives teams flexibility to work with country-specific program structures, additional service areas, or bespoke indicator sets.

This option is useful when:

- the program area goes beyond the Countdown MNCH scope,
- users need a country-defined mapping structure,
- the extraction supports another analytical workflow outside the standard harmonized template.

<Callout type="info">
  **Flexible mapping:** Choose the Custom option when you need to prepare data for other health programs, local reporting needs, or non-Countdown analytical pipelines.
</Callout>

<Callout type="warning">
  **Important mapping choice:** Use Countdown mapping when standardization is the goal. Use Custom mapping when flexibility matters more than alignment to the embedded Countdown MNCH structure.
</Callout>

<a id="step-4-download-data-with-minimal-repeat-requests"></a>

## Step 4: Download data with minimal repeat requests

Once login, metadata retrieval, and mapping are complete, the extractor can begin data download. Instead of relying on a single large DHIS2 request, the extractor is designed to handle extraction in smaller and more manageable operations.

This has several advantages:

- it lowers the risk of timeouts for large requests,
- it reduces the chance of incomplete downloads,
- it allows previously completed work to be reused,
- it supports a more efficient download process after metadata has already been cached.

Because metadata has already been prepared and cached, the extractor can focus on the data values themselves rather than repeatedly reloading the full structural context before each extraction.

<Callout type="warning">
  **Large extractions still need planning:** Even with caching and chunked processing, very large downloads may take time depending on server performance, user permissions, internet stability, and the size of the requested organisation unit and period ranges.
</Callout>

## Step 5: Resume and reuse cached work

Caching is not only useful for metadata. It also helps the extractor preserve progress and reduce unnecessary repeated work where extraction tasks are interrupted or revisited.

This improves reliability in practical field conditions, especially when:

- internet connectivity is unstable,
- DHIS2 response times vary,
- analysts need to rerun the same extract after refining mappings,
- teams want a reproducible process with less manual repetition.

<Callout type="info">
  **Tip:** If you are testing mappings, validate the metadata and structure first using a smaller extraction range before launching a full national, multi-year download.
</Callout>

## Recommended workflow

For most users, the best sequence is:

1. Log in to the correct DHIS2 instance.
2. Cache and review metadata.
3. Choose either the Countdown or Custom mapping option.
4. Validate the mapping before a large extraction.
5. Run the extraction and reuse cached metadata where possible.

## Summary

The CD2030 DHIS2 Data Extractor is built to make DHIS2 extraction more systematic and dependable. It does this by combining authenticated access, metadata caching, structured mapping, and more efficient download behavior.

Its two mapping modes support both standardized Countdown MNCH extraction and more flexible custom program workflows, while local caching helps reduce repeated metadata downloads and keeps the extraction process lighter and more resilient.
